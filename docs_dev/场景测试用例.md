# 测试用例

### 一、 基础功能验证（冒烟测试）

| 用例ID | 测试场景 | 前置条件 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- | :--- |
| **TC-01** | 空数据同步 | A和B均为空 | 启动同步工具 | 同步完成，B库无数据，日志显示无键迁移 |
| **TC-02** | 全量同步基础 | A有10个String键值 | 启动同步（首次运行） | B库存在相同10个键，值完全一致 |
| **TC-03** | 增量同步基础 | 全量同步完成 | 在A中新增、更新、删除各5个键 | B库实时（或准实时）同步变更，最终数据一致 |

### 二、 数据结构覆盖测试（核心数据类型）

| 用例ID | 数据类型 | 测试动作 | 验证点 |
| :--- | :--- | :--- | :--- |
| **TC-04** | **Hash** | 全量同步HSET，增量同步HSET/HDEL | 字段级别变更是否同步；B中`HLEN`和字段值一致 |
| **TC-05** | **List** | 全量同步LPUSH/RPUSH，增量同步LPOP/RPUSH | 元素顺序是否一致（LINDEX校验） |
| **TC-06** | **Set** | 全量同步SADD，增量同步SADD/SREM | 成员是否完整，差集为空 |
| **TC-07** | **Sorted Set (ZSet)** | 全量同步ZADD，增量同步ZADD/ZREM/ZINCRBY | 成员和分数(Score)是否精确一致 |
| **TC-08** | **Stream** | 全量同步XADD，增量同步XADD | 消息ID和内容是否一致 |
| **TC-09** | **Geo/HyperLogLog/Bitmap** | 写入地理位置、PFADD、SETBIT | 特殊结构数据同步后，查询结果（如GEODIST）一致 |

### 三、 全量同步专项测试

| 用例ID | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **TC-10** | 全量+增量混合 | 启动全量同步过程中，持续向A写入新数据（压力写入） | 全量完成基线后，增量能补全同步期间的新增数据，最终数据完全一致 |
| **TC-11** | 大Key同步 | 同步一个包含10万个字段的Hash或10MB大小的String | 同步不超时、不OOM，B库数据完整 |
| **TC-12** | Key过期同步 | A中设置TTL=10s的Key，5s后启动同步 | B库同步该Key，且剩余TTL与A一致（或转为绝对过期时间，由逻辑决定） |
| **TC-13** | 带过期时间（毫秒PX） | 设置PX精度毫秒的Key | B库的TTL精度误差在可接受范围内（如<1s） |

### 四、 增量同步专项测试（核心）

| 用例ID | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **TC-14** | 命令覆盖 | 增量期间依次执行：SET k1 v1 -> SET k1 v2 -> DEL k1 | B库经历相同状态变更，最终Key不存在 |
| **TC-15** | 过期删除同步 | A中Key自然过期（被动删除）或主动DEL | B库对应的Key同步被删除 |
| **TC-16** | 批量操作（MSET/Pipelining） | A端执行MSET k1 v1 k2 v2 | B库同时更新两个键，一致 |
| **TC-17** | Rename/Renamenx | A执行RENAME oldKey newKey | B库oldKey消失，newKey存在且值正确 |
| **TC-18** | FlushDB/FlushAll**（危险）** | 增量期间A执行FLUSHALL | B库需同步清空（建议添加配置开关，测试其行为是否符合预期） |

### 五、 断点续传专项测试（重点 & 难点）

| 用例ID | 测试场景 | 具体故障注入 | 恢复操作 | 预期结果 |
| :--- | :--- | :--- | :--- | :--- |
| **TC-19** | **工具进程重启（全量阶段中断）** | 全量同步RDB传输到50%时，Kill掉同步工具进程 | 重新启动工具（使用相同的状态文件/偏移量） | 工具能感知全量未完成，**重新触发全量同步**（或支持RDB断点重传），最终数据完整 |
| **TC-20** | **工具进程重启（增量阶段中断）** | 增量同步正常运行，Kill掉进程 | 重新启动工具 | 工具根据存储的 **复制偏移量 (Replication Offset)** 或 **SCAN游标** 继续同步，无数据丢失，无数据重复 |
| **TC-21** | **网络闪断（源端不可达）** | 同步中，使用iptables阻断A到工具的网络 | 网络恢复 | 工具自动重连，基于断点续传恢复，无数据丢失 |
| **TC-22** | **网络闪断（目标端不可达）** | 同步中，阻断到B的网络 | 网络恢复 | 工具重试写入，恢复后数据一致（注意幂等性，避免因重试导致重复写入错误值） |
| **TC-23** | **Redis主从切换/重启（Failover）** | 同步中重启Redis A（或发生主从切换，IP变化） | 工具重连新Master | 工具需重新获取RUNID和偏移量，若偏移量不在新Master缓冲区，需自动降级为全量同步 |
| **TC-24** | **磁盘/状态文件损坏** | 模拟checkpoint文件损坏或丢失 | 启动工具 | 工具检测到状态丢失，主动触发**全量同步**，不丢失业务数据 |

### 六、 数据一致性验证（去重与冲突）

| 用例ID | 测试场景 | 操作步骤 | 预期结果 |
| :--- | :--- | :--- | :--- |
| **TC-25** | **幂等性（重复执行）** | 数据完全一致时，重启同步工具 | 工具不产生脏数据，不重复写入，CPU/内存无异常飙升 |
| **TC-26** | **目标库有预置数据** | B库预先有 k1=v_old，A库有 k1=v_new | 同步后，B库被覆盖为 v_new（符合“镜像”预期）；若B数据不在A中，根据策略（保留或删除）测试 |
| **TC-27** | **数据类型冲突** | A中Key类型为String，B中同名Key类型为Hash | 工具应报错或按配置覆盖，不能导致崩溃 |

### 九、 最终验收标准（Checklist）

在执行完所有测试后，建议增加一个**终态校验脚本**，不依赖工具自带的日志，而是直接比对Redis A和B：

1. **Key数量比对**：`DBSIZE` 或 `INFO keyspace`。
2. **全量扫描比对**：使用 `SCAN` 遍历所有Key，比对 `DUMP` 值的二进制 MD5 或使用 `DEBUG DIGEST`（仅限测试环境）。
3. **TTL比对**：抽样检查1000个带过期时间的Key，TTL误差 < 5秒。

---

### 特别建议给你的“断点续传”增加Mock测试

由于断点续传依赖于**同步状态存储**（如本地文件、MySQL或Redis自身），建议单独增加单元测试/集成测试，模拟 `saveCheckpoint()` 和 `loadCheckpoint()` 函数的正确性，确保序列化后的 `{runId, offset, lastSyncTime}` 能精准恢复。这是该工具最核心的价值所在。
